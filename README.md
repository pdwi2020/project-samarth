# Project Samarth

**Intelligent Agri-Climate Q&A System over live data.gov.in sources**

An end-to-end prototype that enables natural language querying of Indian agricultural and climate data from data.gov.in. Built for policy advisors, researchers, and agricultural planners who need quick insights across multiple government datasets.

## üéØ Problem Statement

India's data.gov.in portal hosts thousands of high-granularity datasets across ministries, but they exist in varied formats and structures. Policymakers and researchers struggle to derive cross-domain insights needed for effective decision-making. This project solves that by providing an intelligent Q&A interface that:

- ‚úÖ Sources data directly from live data.gov.in APIs
- ‚úÖ Handles inconsistent data formats and structures
- ‚úÖ Synthesizes information across agriculture and climate datasets
- ‚úÖ Provides fully traceable, cited answers
- ‚úÖ Works offline with graceful degradation

## üöÄ Features

### Data Integration
- **Live API Integration:** Fetches data from data.gov.in CKAN API with pagination
- **Graceful Fallback:** Automatically uses local samples when API is slow/unavailable
- **Multi-Source Synthesis:** Combines data from Ministry of Agriculture & IMD
- **Schema Normalization:** Handles different column names, formats, and encodings

### Intelligent Query Processing
- **4 Question Types Supported:**
  1. **Compare Rainfall + Crops:** Compare rainfall across states and list top crops
  2. **District Extremes:** Find highest/lowest production districts
  3. **Production Trends:** Correlate crop production with climate patterns
  4. **Policy Arguments:** Generate data-backed policy recommendations

- **NLP Parser Features:**
  - Regex-based intent classification (no external APIs - data sovereignty!)
  - **Fuzzy matching** with Levenshtein distance (handles typos)
  - **Crop synonym resolution** (Rice‚ÜîPaddy, Maize‚ÜîCorn, Millet‚ÜîPearl Millet)
  - Case-insensitive state/crop matching

### Analytics Engine
- Cross-dataset joins (agriculture √ó climate)
- Time-series aggregation and trend analysis
- Production correlation with rainfall patterns
- Automatic ranking and sorting

### Production-Ready
- ‚úÖ **100% Test Success Rate:** All 8 official sample prompts working
- ‚úÖ **Full Traceability:** Every data point cites source dataset
- ‚úÖ **Data Sovereignty:** No external LLM APIs, runs completely offline
- ‚úÖ **Error Handling:** Comprehensive validation and user-friendly messages
- ‚úÖ **CORS Enabled:** Frontend-backend separation with proper security

## System Architecture

- **Data Layer** ‚Äì `backend/app/data_manager.py` loads crop and rainfall datasets via the data.gov.in CKAN API (resource IDs pre-configured) or falls back to curated samples (`data/`).
- **Analytics Layer** ‚Äì `backend/app/analytics.py` contains rule-based reasoning templates that join the datasets to produce multi-table answers with correlation commentary and citations.
- **Fuzzy Matching** ‚Äì `backend/app/fuzzy_match.py` implements Levenshtein distance for typo-tolerant matching and synonym resolution.
- **Question Understanding** ‚Äì `backend/app/question_parser.py` maps the challenge's key question families to structured parameters without requiring a heavyweight LLM.
- **API** ‚Äì FastAPI service (`backend/app/main.py`) exposes `/ask` and `/refresh`, returning structured JSON ready for front-end rendering.
- **Frontend** ‚Äì `frontend/index.html` supplies a lightweight chat card that calls the backend, renders tables, and surfaces citations.

### Backend Setup

```bash

# Create virtual environment

# Set up environment variables (API key for data.gov.in)

```

Backend will be available at: **http://localhost:8000**

## üéØ Key Design Decisions

### Why No External LLMs?
**Decision:** Use regex-based NLP instead of GPT/Claude APIs

**Rationale:**
- ‚úÖ **Data sovereignty:** No data leaves the system
- ‚úÖ **Zero latency:** No network calls for parsing
- ‚úÖ **100% reproducible:** Same input ‚Üí same output
- ‚úÖ **Free:** No API costs
- ‚úÖ **Privacy:** Sensitive queries stay local

**Trade-off:** Limited to 4 pre-defined question patterns vs unlimited flexibility

### Why Graceful Fallback?
**Decision:** Always maintain local CSV samples

**Rationale:**
- ‚úÖ Government APIs are unreliable (slow, rate-limited, downtime)
- ‚úÖ Demo resilience: Works even without internet
- ‚úÖ Development velocity: Don't wait for slow APIs during testing

**Implementation:** Fetch live API ‚Üí fallback to local ‚Üí always return data
